# Пълно ръководство за администратора

Добре дошли в портала за администратори на платформата LavBench. Това ръководство описва подробно всеки аспект от конфигурирането, управлението и поддръжката на платформата, изпълнението на работните сървъри (worker nodes), конфигурирането на правилата за сигурност, диагностиката на изграждането на Docker базирани пясъчници и разработването на персонализирани оценители (Custom Evaluators) за състезания по машинно обучение.

### Първоначална настройка

След разгръщане на платформата, инициализирайте администраторския акаунт чрез скрипта за настройка:

```bash
python backend/setup-admin.py
```

Това генерира първоначален администраторски акаунт, създава master credentials и ги записва във файла `admin_credentials.txt` в основната директория на проекта. Използвайте тези данни на главния уеб интерфейс с активирана отметка **"Sign In as Administrator"**.

> [!IMPORTANT]
> Изпълнявайте `setup-admin.py` само веднъж при нова инсталация. Повторното изпълнение на скрипта ще рестартира администраторския достъп.

## 1. Управление на състезанията и етапите

Админ панелът (`/admin`) е централният контролен пункт за конфигуриране на състезания и управление на техния жизнен цикъл през различни етапи.

### Рамка за етапи

Състезанията в LavBench са структурирани около **Етапи (Stages)** (напр. *Квалификационен етап*, *Полуфинали*, *Финали*). Всеки етап поддържа независими конфигурации:
- **Времеви рамки**: Собствени крайни срокове `start_time` и `end_time` за всеки етап.
- **Разпределение на задачите**: Специфични задачи по машинно обучение, зачислени към съответния етап.
- **Навигация между етапи**: Участниците превключват между активните етапи чрез селектора за етапи в горната лента.

### Състояния на жизнения цикъл

| Състояние | Условие / Индикатор | Поведение на системата и участниците |
| :--- | :--- | :--- |
| **Чернова / Не е започнало (Draft)** | `start_time` > текущото време | Видимо само за администратори и зачислени членове на журито (`JuryChallenge`). Използва се за тестване на набори от данни, скриптове за метрики и бейзлайн решения. |
| **Активно (Active)** | `start_time` ≤ текущо време ≤ `end_time` & `is_frozen=False` | Участниците разглеждат задачите, свалят началния код и изпращат решения. |
| **Гратисен период (Grace Period)** | Непосредствено след `end_time` (Таймерът става оранжев) | Кратък активен буфер след крайния срок, приемащ обработвани или изпратени в последната секунда решения преди затваряне на тръбопровода. |
| **Замразено (Frozen / Спешно)** | `is_frozen=True` | Незабавно спира приемането на решения и заключва класирането при инфраструктурни проблеми, без да променя `end_time`. |
| **Приключило (Ended)** | `end_time` < текущото време | Изпращането е затворено. Автоматичното оценяване спира. Журито извършва ръчни проверки и одити на кода. |
| **Финализирано (Finalized)** | `scores_finalized=True` | Класирането е заключено. Псевдонимите са деанонимизирани. Всяка последваща промяна на оценка изисква текстова обосновка в одит лога. |
| **Архивирано (Archived)** | `is_archived=True` | Режим само за четене. Скрити от профилите на участниците; достъпни само за администратори и жури за исторически анализ. |

> [!NOTE]
> Задаването на `is_frozen=True` позволява незабавно спиране на опашката за оценяване, без да се изкривяват времевите печати на състезанието.

## 2. Персонализиране на Sandbox средата и ресурсни лимити

За предпазване на работните сървъри и осигуряване на равнопоставеност, всяка задача по машинно обучение трябва да дефинира изрични ресурсни лимити и изолация.

### Разпределение на ресурсите

- **Лимит на RAM (MB)**: Дефинира лимита на паметта за Docker контейнера (`--memory`). Ако кодът надвиши този лимит, процесът се прекратява незабавно от Linux kernel Out-Of-Memory (OOM) killer.
- **Лимит на време (Секунди)**: Максимално реално време за изпълнение (wall-clock runtime) в контейнера. Надвишаването му води до статусна грешка `TIMEOUT EXPIRED`.
- **Изисква GPU**: Флаг (boolean), насочващ задачата към специалната опашка с GPU хардуерно ускорение.
- **CPU ядра (`--cpus`)**: Брой CPU ядра за контейнера (конфигурира се чрез `CPU_CORES_PER_TASK` / `GPU_CORES_PER_TASK` във `worker.env`).

### Сигурност и изолация на Sandbox контейнера

Кодът на състезателите се изпълнява в защитен Docker контейнер със следните параметри:

| Параметър | Цел |
| :--- | :--- |
| `--network none` | Напълно деактивира мрежата — предотвратява изтичане на данни и външни изтегляния. |
| `--cap-drop ALL` | Премахва всички Linux възможности — блокира raw sockets, `mount()`, `ptrace()` и ескалация на привилегии. |
| `--read-only` | Монтира основната файлова система като read-only — кодът не може да променя системни библиотеки. |
| `--no-new-privileges` | Предотвратява ескалация на привилегии чрез SUID бинарни файлове. |
| `--tmpfs /tmp:noexec,nosuid,size=128m` | Осигурява временна директория в паметта с ограничен размер без право на изпълнение на файлове. |
| `--memory-swap` = RAM лимит | Деактивира swap паметта — предотвратява заобикаляне на лимитите за RAM. |
| `--pids-limit 64` | Ограничава общия брой процеси за предотвратяване на fork бомби. |
| `--ulimit nofile=256:256` | Ограничава броя отворени файлови дескриптори. |

## 3. Грешки при изграждане на Docker контейнери (Build Errors)

### Архитектура на процеса на изграждане

Всеки изолиран пясъчник (sandbox) за дадена задача се базира на персонализирано Docker изображение. Процесът на изграждане протича в четири последователни фази на работния сървър (worker node):

```
[1. Базово изображение] ──> [2. APT пакети] ──> [3. Pip библиотеки] ──> [4. HF кеш & ресурси]
```

1. **Базово изображение (Base Docker Image)**: Изтегляне на базовото изображение (напр. `pytorch/pytorch:2.6.0-cuda12.6-cudnn9-runtime` или `python:3.10-slim`).
2. **Системни APT пакети**: Инсталиране на Ubuntu пакети (напр. `build-essential`, `ffmpeg`, `libglib2.0-0`, `libsndfile1`), зададени в полето `apt_packages`.
3. **Python Pip зависимости**: Генерация на `requirements.txt` от полето `pip_requirements` и изпълнение на `pip install --no-cache-dir -r requirements.txt`.
4. **Офлайн предварително изтегляне (HuggingFace Assets & Task Files)**: Изтегляне на модели (`hf_models`) и набори от данни (`hf_datasets`) в локалната кеш директория (`hf_cache`), както и копиране на `evaluator_script.py` и ресурсни файлове на задачата в контейнерния образ.

Генерираното изображение се именува по шаблона `lavbench_task_{task_id}`. Системата изчислява SHA-256 хеш (`_config_hash`) от всички параметри на задачата. Ако конфигурацията не е променена и изображението вече съществува, изграждането се пропуска автоматично.

### Класификация на грешките при изграждане (Build Errors)

При възникване на проблем по време на изграждането, задачата се маркира със статус за грешка, а подробният лог се записва в полето `build_error` на модела `Task`. Основните категории грешки включват:

#### 1. Невалидно базово изображение (Invalid Base Image)
* **Причина**: Посоченият таг или име на изображението не съществува в Docker Hub/регистъра (напр. `pytorch/pytorch:nonexistent-tag`) или липсват права за достъп (HTTP 404 / 401 Unauthorized).
* **Симптом в лога**: `Error response from daemon: manifest for ... not found` или `pull access denied`.
* **Решение**: Проверете точния таг на изображението в Docker Hub и го коригирайте в настройките на задачата.

#### 2. Несъществуващ или несъвместим APT пакет (APT Resolution Failure)
* **Причина**: Печатна грешка в името на Ubuntu пакета (напр. `libglib2.0-devs` вместо `libglib2.0-dev`) или пакет, който не съществува в базовия репозиторий.
* **Симптом в лога**: `E: Unable to locate package ...` или `E: Package '...' has no installation candidate`.
* **Решение**: Коригирайте списъка с APT пакети в Админ панела. Използвайте имена на пакети, съвместими с дистрибуцията на базовото изображение.

#### 3. Pip конфликти във версиите и компилационни грешки (Pip Dependency & Compilation Conflicts)
* **Причина**: Несъвместими ограничения във версиите на Python библиотеките (напр. `transformers==4.30.0` с `torch==1.8.0`), липсващи прекомпилирани binary wheels за текущата Python версия или липса на C/C++ заглавни файлове (`gcc`/`g++`).
* **Симптом в лога**: `ERROR: Cannot install ... because these package versions have conflicting dependencies`, `Failed building wheel for ...` или `fatal error: Python.h: No such file or directory`.
* **Решение**: Проверете съвместимостта на версиите в `pip_requirements`. Добавете необходимите компилационни APT пакети (`build-essential`, `python3-dev`).

#### 4. Timeout и грешки при изтегляне на Hugging Face ресурси (HF Pre-Fetch Timeout / Auth Failures)
* **Причина**: Прекъсване на мрежовата връзка по време на сваляне на големи тегла, надвишаване на съответния мрежов timeout (`WORKER_DOWNLOAD_TIMEOUT`), липса на valid `hf_api_key` за частни или защитени (gated) модели (напр. `meta-llama/Llama-3-8B`), или надвишаване на HF API rate limits.
* **Симптом в лога**: `RepositoryNotAccessibleError`, `401 Client Error: Unauthorized for url`, `ReadTimeoutError` или `GatedRepoError`.
* **Решение**: Посочете валиден `hf_api_key` с достъп до модела в настройките на задачата, или увеличете таймаута при изтегляне.

#### 5. Недостиг на памет или дисково пространство при изграждане (Build OOM / Disk Exhaustion)
* **Причина**: Недостиг на RAM памет по време на компилиране на тежки Python пакети (напр. `flash-attn`, `vllm`), изчерпване на дисковото пространство на хост машината под `MIN_BUILD_DISK_GB` (минимум 5 GB свободно пространство в `TASK_IMAGES_DIR`).
* **Симптом в лога**: `c++: fatal error: Killed process (program cc1plus)`, `Insufficient disk space for Docker build` или `No space left on device`.
* **Решение**: Освободете дисково пространство на работния сървър чрез почистване на стари контейнери (`docker system prune`), увеличете паметта на работната машина или използвайте предварително компилирани wheels.

### Преглед на статуса на изграждане и логовете

Администраторите разполагат с два независими канала за проследяване на изграждането:

#### В Уеб Админ Панела:
1. Навигирайте до **Админ Панел** → **Задачи (Tasks)** → изберете съответната задача.
2. Секцията **Статус на Docker изображението** показва текущото състояние (`Ready`, `Building`, `Failed`).
3. При възникване на грешка се визуализира червен панел **Build Error**, съдържащ точната причина за провала и последната налична линия от лога.

#### В логовете на Celery Работните Сървъри:
1. Отворете логовете на работния сървър (Worker log):
   ```bash
   journalctl -u lavbench-worker -f -n 200
   # Или при работа в Docker Compose:
   docker logs -f lavbench_celery_worker
   ```
2. Търсете записи с префикс `[build lavbench_task_<id>]`. Всеки ред от `docker build` се стриймва в реално време:
   ```text
   INFO [build lavbench_task_018f3a2b] Step 3/5 : RUN pip install --no-cache-dir -r /tmp/requirements.txt
   ERROR [build lavbench_task_018f3a2b] Error: ERROR: No matching distribution found for numpy==9.9.9
   ```
3. Неуспешните задачи за оценяване се регистрират и в Опашката за неуспешни задачи (`/api/admin/dead-letters`).

### Отстраняване на грешки (`ERR_IMAGE_BUILD_FAILED`), изчистване на кеша и повторно стартиране

Ако изграждането се провали със статус грешка `ERR_IMAGE_BUILD_FAILED`:

1. **Коригирайте грешката в конфигурацията**: Променете невалидните APT пакети, Pip библиотеки или базово изображение в формата за редакция на задачата.
2. **Изчистете заседналите заключения (Build Locks)**:
   Ако изграждането е прекъснато неочаквано (напр. при рестарт на worker сървъра), в Redis може да остане заседнало заключване (`build_lock`). Изчистете го чрез бутона **Force Clear Build Lock** в админ панела или чрез функцията `clear_build_lock(task_id)`.
3. **Почистете невалидните междинни Docker кешове**:
   Изпълнете на работния сървър за премахване на незавършени Docker слоеве:
   ```bash
   docker image prune -f
   ```
4. **Стартирайте повторно изграждането (Trigger Rebuild)**:
   При запазване на променената задача, платформата автоматично изпраща нотификация по канала Redis `task_rebuild`. Работните сървъри улавят съобщението, анулират стария кеш хеш и започват ново изграждане на изображението `lavbench_task_{task_id}`.

> [!TIP]
> Можете да форсирате повторно изграждане и без промяна на конфигурацията, като кликнете върху бутона **Rebuild Container Image** в картата на задачата.

## 4. Предварително изтегляне от Hugging Face и офлайн ресурси

Поради стриктната сигурност на sandbox контейнерите (`--network none`), моделите и наборите от данни не могат да се теглят през интернет по време на изпълнение на кодовете на състезателите.

### Конфигуриране на ресурси

Администраторите дефинират предварително изтегляните Hugging Face ресурси за всяка задача:
- **`hf_datasets`**: Списък с ID на набори от данни (напр. `["glue", "sst2"]`).
- **`hf_models`**: Списък с ID на репозитории за модели (напр. `["bert-base-uncased", "distilbert/distilbert-base-uncased"]`).

```json
{
  "hf_datasets": ["glue", "sst2"],
  "hf_models": ["bert-base-uncased", "distilbert/distilbert-base-uncased"]
}
```

### Механизъм на работа

1. По време на изграждане на Docker изображението, фоновата услуга извиква `load_dataset` и `snapshot_download` от библиотеките `datasets` и `huggingface_hub`.
2. Ресурсите се записват в директорията `hf_cache` на съответната задача (`/TASK_IMAGES_DIR/task_{id}/hf_cache`).
3. При изграждане на Dockerfile командата `COPY hf_cache /hf_cache` вгражда данните директно вътре в Docker изображението.
4. При стартиране на решенията, скриптовете на състезателите използват библиотеки `transformers` или `datasets` и зареждат моделите напълно офлайн.

## 5. AST сигурност на кода и динамична машина за метрики

### Валидация на кода чрез AST преди изпълнение

Преди всяко изпратено решение да се добави в опашката за изпълнение, сървърът извършва статичен анализ на сигурността (AST):

1. **Премахване на IPython Magic команди**: Регулярен израз автоматично премахва IPython/Jupyter shell команди (`%matplotlib inline`, `!pip install`, `%timeit`) преди извършване на AST анализа.
2. **Верификация на импортираните модули**:
   - `banned_imports`: Списък със забранени Python модули (напр. `os, sys, subprocess, socket, requests, urllib, shutil`).
   - `whitelisted_imports`: Ограничителен списък с позволени модули (при активиран строг режим).
3. **Запазване на квотата**: Ако AST валидацията открие синтактична грешка или забранен импорт, решението получава статус `Failed`, показват се детайли за грешката, а **дневната квота за опити на състезателя НЕ се намалява**.

### Динамична схема за метрики

Оценъчните метрики се дефинират за всяка задача чрез JSON схема с тегла за публичния и частния резултат.

**Пример за конфигурация с множество метрики:**
```json
{
  "accuracy": { "weight": 0.5, "higher_is_better": true },
  "f1": { "weight": 0.5, "higher_is_better": true }
}
```

### Поддържани категории метрики (44 метрики в 12 категории)

LavBench включва 44 вградени метрики за оценяване в 12 предметни области:

| Категория | Ключове на метрики | Описание / Основно приложение |
| :--- | :--- | :--- |
| **Класификация** | `accuracy`, `f1`, `precision`, `recall`, `cohen_kappa`, `matthews_corrcoef` | Оценка на дискретни прогнози. |
| **Вероятностни** | `auc_roc`, `logloss`, `brier_score` | Оценка на непрекъснати вероятностни оценки. |
| **Регресия** | `rmse`, `mse`, `mae`, `r_squared`, `mape`, `median_ae` | Измерване на грешката при непрекъснати цели. |
| **NER / Секвенции** | `seqeval_f1`, `seqeval_precision`, `seqeval_recall` | Извличане на именовани обекти и токени. |
| **Генеративен NLP** | `bleu`, `rouge`, `rouge_l`, `meteor`, `chrf`, `ter` | Машинен превод, резюмиране и генерация на текст. |
| **QA Извличане** | `exact_match`, `f1` (word-overlap) | Текстово съвпадение при въпроси и отговори. |
| **Засичане на обекти** | `map_50`, `map_75`, `map_50_95`, `recall` (box recall) | Оценка на ограничителни рамки (bounding boxes IoU / mAP). |
| **Сегментация** | `mean_iou`, `dice`, `pixel_accuracy` | Семантични маски и пикселни съвпадения. |
| **Ключови точки** | `oks`, `pck` | Оценка на позите на обекти и ключови точки. |

## 6. Логика и архитектура на персонализирани оценители (Custom Evaluators)

### Архитектура и предназначение

За специфични задачи по машинно обучение, при които стандартните scikit-learn метрики не са достатъчни (напр. оптимизация на графи, специализиран аудио анализ, комплексeн домейн-специфичен score), администраторите могат да прикачат персонализиран Python скрипт (`evaluator.py`).

Скриптът се изпълнява директно в процеса за оценяване и замества стандартната логика за изчисляване на метриките.

### Задължителна структура и сигнатура на функцията

Всеки персонализиран скрипт (`evaluator.py`) **ТРЯБВА** да дефинира следните четири глобални променливи и основна функция за оценяване `evaluate()`:

```python
import numpy as np
import pandas as pd

# 1. Глобален идентификатор на главната метрика (задължителен)
METRIC_NAME = "custom_score"

# 2. Дефиниция на очакваните колони в submission.parquet на участника (за админ визуализация)
SUBMISSION_COLUMNS = [
    {"name": "id", "type": "string"},
    {"name": "prediction", "type": "float"},
]

# 3. Дефиниция на очакваните колони в labels.parquet (скритите верни отговори)
LABELS_COLUMNS = [
    {"name": "id", "type": "string"},
    {"name": "target", "type": "float"},
]

# 4. По подразбиране опции, които се подават в аргумента 'options'
EVALUATOR_OPTIONS = {
    "tolerance": 0.05,
    "penalty_weight": 0.1,
}


def evaluate(predictions_df, labels_df, options=None):
    """Персонализирана функция за изчисляване на оценките.

    Parameters:
        predictions_df (pd.DataFrame): DataFrame от submission.parquet на участника.
        labels_df (pd.DataFrame): DataFrame от labels.parquet (ground truth).
        options (dict, optional): Опции от метричната конфигурация / options_schema.

    Returns:
        dict[str, float]: Речник с изчислените стойности на метриките.
                          Ключът 'custom_score' (съвпадащ с METRIC_NAME) е задължителен.
    """
    if options is None:
        options = {}

    tolerance = options.get("tolerance", 0.05)
    penalty_weight = options.get("penalty_weight", 0.1)

    # 1. Защита и проверки за съществуване на колоните
    if "id" not in predictions_df.columns or "prediction" not in predictions_df.columns:
        raise ValueError("submission.parquet трябва да съдържа колони 'id' и 'prediction'.")

    if "id" not in labels_df.columns or "target" not in labels_df.columns:
        raise ValueError("labels.parquet трябва да съдържа колони 'id' и 'target'.")

    # 2. Подравняване на данните по ID
    merged = pd.merge(predictions_df, labels_df, on="id", how="inner", suffixes=("_pred", "_true"))
    if merged.empty:
        return {METRIC_NAME: 0.0, "secondary_metric": 0.0}

    # 3. Обработка на липсващи стойности (NaN / Null)
    pred_values = pd.to_numeric(merged["prediction"], errors="coerce").fillna(0.0).to_numpy()
    true_values = pd.to_numeric(merged["target"], errors="coerce").fillna(0.0).to_numpy()

    # 4. Изчисление на персонализирана метрика
    abs_errors = np.abs(pred_values - true_values)
    within_tolerance = (abs_errors <= tolerance).astype(float)
    accuracy_rate = float(np.mean(within_tolerance))

    mean_penalty = float(np.mean(abs_errors) * penalty_weight)
    final_score = max(0.0, float(accuracy_rate - mean_penalty))

    return {
        METRIC_NAME: final_score,
        "secondary_metric": float(np.mean(abs_errors)),
    }
```

> [!IMPORTANT]
> При качване на скрипта в Админ панела, backend услугата извършва AST валидация. Ако някой от променливите `METRIC_NAME`, `SUBMISSION_COLUMNS` или `LABELS_COLUMNS` липсва или функцията `evaluate` не е дефинирана с точната сигнатура, качването се отхвърля с грешка HTTP 400 Bad Request.

### Настройка на `metrics_config` и `options_schema`

След качване на персонализиран оценител, конфигурирайте полето **Metrics Config** на задачата в Админ панела, за да съвпадне с върнатите от скрипта речникови ключове:

```json
{
  "custom_score": { "weight": 0.8, "higher_is_better": true },
  "secondary_metric": { "weight": 0.2, "higher_is_better": false }
}
```

Параметрите в `EVALUATOR_OPTIONS` могат динамично да се пренаписват от полето `evaluator_options_schema` в задачата, с цел промяна на коефициенти без редактиране на изходния Python код.

### Качване на `evaluator.py` в Админ Панела

1. Отидете в **Админ Панел** → **Задачи (Tasks)** → изберете или създайте задача.
2. В секцията **Персонализиран оценител (Custom Evaluator)**:
   - Кликнете **Upload Evaluator Script (.py)** и изберете локалния скрипт `evaluator.py`.
   - Или поставете Python кода директно в текстовия редактор `custom_eval_code`.
3. Записването автоматично извлича `METRIC_NAME`, записва кода в базата данни и го експортира във файла `evaluator_script_path` за работните сървъри.

### Тестване чрез референтни baseline soluzioni (`is_baseline=True`)

За да гарантирате, че персонализираният оценител функционира правилно преди старта на състезанието:

1. Администраторът или организаторът подготвя референтно Jupyter Notebook решение.
2. При изпращане на решението от уеб интерфейса, маркирайте отметката **"Mark as Baseline Solution"** (`is_baseline=True`).
3. Решението се оценява с висок приоритет без намаляване на потребителските квоти.
4. Проверете изходните стойности в класирането — успешно намерените стойности за `METRIC_NAME` и `secondary_metric` потвърждават изправността на скрипта.

### Защита срещу изключения, липсващи колони и NaN стойности

Персонализираните скриптове трябва да са изключително устойчиви на грешки в кодовете на участниците. Следвайте следните добри практики:
- **Защита срещу липсващи колони**: Винаги проверявайте дали очакваните колони съществуват в `predictions_df`. Ако участникът е записал грешно име на колона, хвърлете `ValueError` с ясно съобщение, което да се покаже в лога на решението.
- **Обработка на NaN и Infinite стойности**: Използвайте `pd.to_numeric(..., errors='coerce')` и `.fillna(0.0)` или `np.nan_to_num()`, за да предотвратите срив на изчисленията при предадени `NaN` или `Inf` стойности.
- **Валидиране на типовете**: Каствайте финалните резултати до стандартни Python типове (`float(...)`), тъй като NumPy float64 обекти могат да причинят проблеми при JSON сериализация.

## 9. Автентикация на работните сървъри, CLI настройка и SSE телеметрия

### Ed25519 Асиметрична автентикация

Работните сървъри комуникират сигурно с уеб платформата чрез асиметрични Ed25519 ключове:
- Работният сървър подписва заявките със своя частен ключ.
- Токените се предават в HTTP хедъра `X-Worker-Token`.
- Backend услугата верифицира подписа спрямо публичния ключ на съответния worker.

### Интерактивна CLI настройка (`make setup-worker`)

```bash
# 1. Изтегляне на конфигурационния шаблон:
scp user@server:~/lavbench/worker.env .

# 2. Стартиране на съветника за настройка:
make setup-worker
```

Съветникът ви превежда през:
1. **Режим на Sandbox**: Docker контейнер (препоръчително) или локална среда.
2. **Тип на Worker**: `evaluation` (изпълнява решения), `internal` (системни задачи/backups) или `both`.
3. **GPU Конфигурация**: Избор на GPU устройства чрез `nvidia-smi`.
4. **CPU ядра**: Лимити за CPU ядра на контейнер.
5. **Concurrency**: Автоматично изчисление на паралелните слотове.

```bash
# Стартиране на worker процеса:
make deploy-worker

# Редактиране на настройките:
make edit-worker
```

> [!WARNING]
> При стартиране на evaluation worker на основния уеб сървър, запазете поне 2 CPU ядра свободни за Gunicorn, PostgreSQL и Redis.

### SSE Телеметрия на клъстера

Състоянието на работните сървъри се предава в реално време към администраторската навигационна лента чрез Server-Sent Events (SSE):
- **Индикатор за статус**: Зелена значка (свързани и здрави сървъри) / Червена значка (прекъсната връзка).
- **Модален прозорец за клъстера**: Показва CPU ядра, RAM, GPU модел, VRAM памет и активни слотове за изпълнение.
- **Опашка за неуспешни задачи (`/api/admin/dead-letters`)**: Показва окончателно провалилите се Celery задачи с пълен stack trace за диагностика.

### Одит система за проследяемост

Всяка чувствителна административна операция се записва перманентно в одит лога (таблица `AuditLog`).

- **Проследявани операции**:
  - Създаване, промяна или изтриване на състезания, етапи и задачи.
  - Промени на състоянието (`finalize`, `archive`, `freeze`).
  - Регистриране на потребители, CSV импортиране и нулиране на пароли.
  - Дисквалификации (`is_disqualified`) и бейзлайн флагове (`is_baseline`).
  - Ръчни редакции на оценки.

### Основания за промяна на оценки след финализиране

При промяна на оценка след финализиране на състезанието (`scores_finalized=True`):

1. Интерфейсът показва задължителен модален прозорец **Audit Justification Modal**.
2. Администраторът въвежда текстово основание (напр. *"Корекция на оценка след уважена контестация за формат на данните"*).
3. Системата записва неизтриваем запис, съдържащ:
   - ID и Роля на потребителя
   - ID на състезателя и решението
   - Първоначална vs Нова оценка
   - ISO Времеви печат
   - Текстово основание / Причина

Прегледът на одит лога е достъпен по всяко време през **Админ Панел** → **Audit Logs**.
