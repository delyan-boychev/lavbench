# Пълно ръководство за администратора

Добре дошли в портала за администратори на платформата LavBench. Това ръководство описва подробно всеки аспект от конфигурирането, управлението и поддръжката на платформата.

### Първоначална настройка

След разгръщане на платформата, създайте администраторския акаунт:

```bash
python backend/setup-admin.py
```

Това изтрива и пресъздава базата данни, генерира произволно администраторско потребителско име и master key, и ги записва в `admin_credentials.txt` в основната директория на проекта. Използвайте тези данни на страницата за вход с отметка "Sign In as Administrator".

> Изпълнявайте това само веднъж на чиста база данни. Повторното изпълнение изтрива всички данни.

## 1. Управление на жизнения цикъл на състезанието

Админ панелът (`/admin`) е вашият централен хъб за създаване на състезания и управление на техните етапи.

### Състояния на състезанието

- **Чернова / Не е започнало (Draft / Not Started):** `start_time` > текущото време. Видимо само за администратори и жури. Използвайте тази фаза за безопасно тестване на конфигурации, скриптове за метрики и ресурсни лимити.
- **Активно (Active):** `start_time` <= текущо време <= `end_time` и `is_frozen=False`. Участниците могат да разглеждат задачите и да изпращат решения. Функцията **Гратисен период (Grace Period)** позволява валидни опити в последната секунда непосредствено след изтичането на `end_time`.
- **Замразено (Frozen / Спешно ръчно замразяване):** Ако `is_frozen=True`. Това ви позволява незабавно да спрете изпращането на решения и да замразите класирането при проблеми с инфраструктурата или критични бъгове, _без_ да се налага ръчно да удължавате крайния срок (`end_time`).
- **Приключило (Ended):** `end_time` < текущото време. Изпращането на решения е затворено, но резултатите все още не са финализирани. Журито може да извърши одити и да добави ръчни оценки.
- **Финализирано (Finalized):** `scores_finalized=True`. Класирането е заключено и самоличностите (псевдонимите) на участниците са деанонимизирани. Редакциите на оценките след финализиране са позволени, но строго изискват запис с обосновка в одитния лог.
- **Архивирано (Archived):** `is_archived=True`. Режим само за четене (Read-only). **Много важно**: архивираните състезания са напълно скрити от профилите на участниците. Само администратори и жури запазват достъп до статистиките им.

## 2. Персонализиране на sandbox и лимити на ресурсите

За да се предпази клъстерът за изпълнение и да се гарантира честна игра, всяка задача трябва да бъде изрично конфигурирана.

### Разпределение на ресурсите

- **Лимит на RAM:** (В MB) Задава строг `-m` лимит на Docker контейнера. Ако обработката на данните или размерът на модела на участника надвиши този лимит, процесът се прекратява незабавно със сигнал Out-Of-Memory (OOM).
- **Лимит на време:** (В секунди) Максималното реално време за изпълнение (wall-clock runtime) на Celery процеса. Надвишаването му води до грешка Timeout.
- **Изисква GPU:** Флаг (boolean), който насочва задачата за изпълнение специфично към Celery опашката за високопроизводителен GPU инференс.

### Сигурност на sandbox средата

Всяко решение се изпълнява в укрепен Docker контейнер със следните ограничения:

| Ограничение                            | Цел                                                                                                    |
| -------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| `--network none`                       | Блокира мрежовия достъп — без изтичане на данни или изтегляния                                         |
| `--cap-drop ALL`                       | Премахва всички Linux възможности — без raw sockets, `mount()`, `ptrace()` или ескалация на привилегии |
| `--read-only`                          | Файловата система е само за четене — кодът не може да променя системни файлове                         |
| `--no-new-privileges`                  | Предотвратява ескалация чрез suid бинарни файлове                                                      |
| `--tmpfs /tmp:noexec,nosuid,size=128m` | Временна файлова система в паметта (tmpfs) с ограничен размер — не може да запълни диска               |
| `--memory-swap` = RAM лимит            | Деактивира swap — заобикаляне на RAM лимита е невъзможно                                               |
| `--pids-limit 64`                      | Предотвратява fork бомби                                                                               |
| `--ulimit nofile=256:256`              | Ограничава отворените файлови дескриптори                                                              |
| `--cpus 2`                             | Ограничава процесора до 2 ядра                                                                         |

### Настройка на потребителско Docker изображение

- **Базово изображение (Base Image):** Посочете валидно публично Docker изображение от регистър (напр. `pytorch/pytorch:2.0.1-cuda11.7-cudnn8-runtime`).
- **APT пакети:** Списък с Ubuntu пакети, разделени със запетая (напр. `libglib2.0-0, build-essential`).
- **Pip зависимости:** Дефинирайте стандартни Python библиотеки, необходими за изпълнението на задачата.

> [!NOTE]
> Инфраструктурата на работните сървъри автоматично изгражда тези зависимости динамично при първото изпълнение и кешира готовия контейнер (sandbox).

## 3. Конфигурация на метриките и системата за правила

### Динамична схема за метрики

JSON конфигурация определя как се усредняват финалните публични и частни резултати.

**Пример: Оптимизиране на Точност (Accuracy) и F1 Score**

```json
{
  "accuracy": { "weight": 0.5, "higher_is_better": true },
  "f1": { "weight": 0.5, "higher_is_better": true }
}
```

**Пример: Регресионна задача с множество метрики**

```json
{
  "rmse": { "weight": 0.6 },
  "mae": { "weight": 0.3 },
  "r_squared": { "weight": 0.1 }
}
```

### Поддържани категории метрики

Машината за оценяване поддържа ~70 метрики в 12 категории задачи:

| Категория                   | Ключове на метрики                                                                   |
| --------------------------- | ------------------------------------------------------------------------------------ |
| **Класификация**            | `accuracy`, `f1`, `precision`, `recall`, `cohen_kappa`, `matthews_corrcoef`          |
| **Вероятностни**            | `auc_roc`, `logloss`, `brier_score`                                                  |
| **Регресия**                | `rmse`, `mse`, `mae`, `r_squared`, `mape`, `median_ae`                               |
| **Поредни етикети (NER)**   | `seqeval_f1`, `seqeval_precision`, `seqeval_recall`                                  |
| **Генеративен NLP**         | `bleu`, `rouge`, `rouge_l`, `meteor`, `bertscore`, `chrf`, `ter`                     |
| **QA Извличане**            | `exact_match`, `f1` (съвпадение на думи)                                             |
| **Засичане на обекти**      | `map_50`, `map_75`, `map_50_95`, `recall` (box recall)                               |
| **Сегментация**             | `mean_iou`, `dice`, `pixel_accuracy`                                                 |
| **Ключови точки**           | `oks`, `pck`                                                                         |
| **Качество на изображение** | `psnr`, `ssim`, `fid`, `is`, `clip_score`, `lpips`, `niqe`                           |
| **Качество на аудио**       | `snr`, `mel_lsd`, `si_sdr`, `nisqa`, `pesq`                                          |
| **Клъстеризация**           | `adjusted_rand_index`, `normalized_mutual_info`, `adjusted_mutual_info`, `v_measure` |
| **Извличане**               | `ndcg_k`, `recall_k`, `mrr`                                                          |

> [!NOTE]
> `f1` и `recall` автоматично избират клон на изчисление според типа данни: низови стойности → QA word-overlap; списъци от речници → box recall (засичане на обекти); числа → sklearn класификация.

### AST валидация и правила преди изпълнение

Преди дадено решение да достигне до Celery опашката, то преминава през строг статичен анализ на сигурността (AST):

- **Автоматично премахване на магически команди:** Jupyter `%` и `!` команди (напр. `%matplotlib inline`, `!pip install`) се изтриват автоматично чрез регулярен израз преди AST анализа. Те нямат ефект в sandbox средата.

### Забранени библиотеки (Banned Imports)

Дефинирайте модули (като `os, sys, subprocess, requests, socket`), които са забранени за импортиране. Те се конфигурират за всяка задача поотделно.

### Как работи оценяването

Участниците изпращат Jupyter notebook файлове. Техният код се изпълнява се в изолиран Docker sandbox и записва файл **`submission.parquet`** с техните прогнози (predictions). Системата го сравнява със скрития файл **`labels.parquet`** (качен към задачата), за да изчисли резултатите.

- Кодът на участника трябва да генерира `submission.parquet` с колона `id` и колони с прогнози.
- `labels.parquet` съдържа реалните етикети (ground truth) — качете го при създаването на задачата.
- Полето `metrics_config` на задачата дефинира кои метрики да се изчислят и с каква тежест.
- `public_eval_percentage` контролира разпределението между публичния и частния дял (public/private split).

## 5. Управление на резервни копия (Backups)

Платформата разполага с протокол за автоматично архивиране, за да предотврати загуба на данни. Всички резервни копия включват пълен дъмп на PostgreSQL и директорията `uploads/`, компресирани в `.tar.gz` файл.

### Видове резервни копия

| Тип             | Честота                                                                      | Съхранение              | Местоположение                                |
| --------------- | ---------------------------------------------------------------------------- | ----------------------- | --------------------------------------------- |
| **Автоматични** | На всеки **20 минути**, когато има активни състезания; на **6 часа** в покой | Последните **6**        | `auto_YYYYMMDD_HHMMSS.tar.gz` в `/backups/`   |
| **Ръчни**       | При поискване чрез бутона "Създай резервно копие сега"                       | Не се трият автоматично | `manual_YYYYMMDD_HHMMSS.tar.gz` в `/backups/` |

### Интерфейс за управление на резервните копия

Отидете в **Админ панел** → раздел **Резервни копия** (Backups):

- Преглед на списъка с всички автоматични и ръчни архиви с размери на файловете и времеви печати.
- **Създай резервно копие сега (Force Backup Now)** — създава незабавен ръчен архив. Показва индикатор за зареждане чрез SSE.
- **Изтегли (Download)** — изтегляне на избран архивен файл.
- **Изтрий (Delete)** — изтриване само на ръчни архиви (автоматичните не могат да се трият ръчно, връща 403).

### Правила за съхранение

- **Автоматични**: Пазят се само 6-те най-скорошни. Най-старите се трият автоматично.
- **Ръчни**: Никога не се трият автоматично. Премахват се само чрез бутона за изтриване.

## 6. Мониторинг и диагностика

### Ендпойнт за проверка на състоянието (Health Endpoint)

`GET /api/health` — проверява връзката с базата данни. Връща:

```json
{ "status": "ok", "database": "connected" }
```

Връща 503, ако базата данни е недостъпна. Използва се от Docker health checks и load balancers.

### Опашка за неуспешни задачи (Dead Letter Queue)

Когато дадена Celery задача за оценяване се провали окончателно (всички опити за рестартиране са изчерпани), тя се записва в Redis dead letter queue. Можете да я прегледате чрез:
`GET /api/admin/dead-letters` (само за администратори).

Всеки запис показва ID на решението, ID на задачата, ID на състезанието, време на провала и съобщение за грешка. Полезно за дебъгване на системни проблеми с оценяването.

### Одит лог (Audit Logs)

Платформата поддържа подробна история на всички административни действия (audit trail). Това е от критично значение за гарантиране на прозрачност и почтеност при управлението на състезатели, оценяване и настройки.

- **Достъп**: Навигирайте до **Админ панел** → раздел **Одит лог** (Audit Logs) — достъпен само за потребители с роля `admin`.
- **Записвани действия**:
  - `create` (създаване), `update` (редакция), `delete` (изтриване) на състезания, етапи и задачи.
  - Действия по финализиране (`finalize`) и архивиране (`archive`).
  - Промени на пароли (индивидуални или масови за цяло състезание).
  - Ръчни корекции на оценки (които задължително изискват въвеждане на текстово основание).
- **Филтриране**: Можете да търсите/филтрирате одит лога по **Тип действие** (Action Type) или да преглеждате само действия, свързани с конкретно състезание.
- **Детайли**: Кликнете на **Виж (View)** до всеки запис, за да прегледате пълните метаданни на действието (payload) в JSON формат, както и посоченото основание/причина (например защо е редактирана оценка).

### Мониторинг на статуса на работните сървъри (Workers)

Значката **Клъстер** (Cluster) в навигационната лента показва състоянието на работните сървъри в реално време чрез SSE:

- Зелено: сървърите са свързани и работят нормално.
- Червено: сървърите са изключени (disconnected).
- Кликнете за отваряне на модален прозорец, показващ спецификациите на всеки worker (CPU ядра, RAM, тип GPU, VRAM, конкурентност (concurrency)).

Работните сървъри са разделени на две категории с цел изолиране на системните задачи от тежките натоварвания по оценяване:

1. **Вътрешен работен сървър (Internal Task Worker)**: Работи в рамките на Docker Compose конфигурацията на уеб сървъра (услуга `celery_worker`). Той изпълнява системни задачи като резервни копия на базата данни, проверки на watchdog и преизчисляване на класации. По подразбиране неговата конкурентност (concurrency) е ограничена до `2` (чрез `CELERY_WORKER_CONCURRENCY`), за да се запазят ресурсите на хоста за Flask, PostgreSQL, Redis и SSE.
2. **Оценяващи работни сървъри (Evaluation Workers)**: Работят директно на хост/отдалечени машини (с CPU или GPU ресурси) за изграждане и стартиране на Docker пясъчници за студентските решения.

#### Стартиране на оценяващи работни сървъри

Препоръчителният начин е чрез интерактивната настройка:

```bash
# Копирайте worker.env от сървъра (генериран от make setup-server):
scp user@server:~/lavbench/worker.env .

# Интерактивна конфигурация (режим, тип, GPU, ядра):
make setup-worker
```

При първи старт скриптът открива наличните CPU и GPU ресурси и ви превежда през:
1. **Режим** — Docker (препоръчително) или локален micromamba
2. **Тип на worker-а** — оценяващ (evaluation), вътрешен (internal) или и двете
3. **GPU ID** — избор кои GPU устройства да се използват (изисква `nvidia-smi`)
4. **CPU ядра на задача** — разпределение за GPU и за CPU evaluation контейнери
5. **Concurrency** — автоматично изчислена (брой GPU + останали CPU слотове)

След това стартирайте от записаната конфигурация:

```bash
make deploy-worker
```

За редактиране на запазената конфигурация:

```bash
make edit-worker    # меню-базиран редактор (тип, GPUs, ядра, concurrency)
```

_Забележка: Ако стартирате оценяващи работни сървъри на основния уеб хост, оставете поне 2 свободни CPU ядра, за да предотвратите недостиг на ресурси за контейнерите на Gunicorn и Postgres._

### Регистрация на спецификациите на сървърите

Когато даден оценяващ работен сървър стартира, той автоматично регистрира своите хардуерни спецификации в Redis. Те се визуализират в таблото за клъстера. Ако Redis е недостъпен, спецификациите на worker-а няма да бъдат регистрирани до следващия рестарт. Вътрешните работни сървъри (internal workers) са изключени от показателите и телеметрията на клъстера.
